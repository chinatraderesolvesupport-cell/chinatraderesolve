# Подключение анализа документов v3.5.9

## Что добавлено

После отправки заявки клиент получает закрытую ссылку дела. На этой странице он может загрузить до двадцати ключевых файлов:

- PDF;
- JPG/JPEG;
- PNG;
- WebP.

Ограничения: не более 8 МБ на один файл и 45 МБ суммарно на одно дело.
Лимит 45 МБ оставляет запас относительно ограничения OpenAI в 50 МБ на совокупность файлов одного запроса.
Дополнительно сервер отклоняет HTTP-запрос загрузки свыше 50 МБ до разбора multipart. В эти 50 МБ входит небольшая служебная обёртка формы, поэтому фактический набор файлов должен оставаться в пределах указанного лимита 45 МБ.

Изображения перекодируются сервером, поэтому EXIF и другие встроенные метаданные удаляются. Файлы сохраняются в той же PostgreSQL-базе, что и дело, и не зависят от временного диска Render.

ИИ формирует предварительный структурированный отчёт:

- индекс готовности доказательств;
- состав и читаемость документов;
- хронологию;
- ключевые доказательства;
- возможные противоречия;
- недостающие материалы;
- факторы риска;
- рекомендуемые следующие шаги.

Индекс готовности показывает только организованность доказательств. Это не вероятность победы и не юридическое заключение.


## Отзывчивость сервера при обработке изображений

Проверка PDF и перекодирование изображений выполняются в рабочих потоках, а не в основном ASGI event loop. Крупный скриншот поэтому не останавливает ответы формы и `/health`. Для защиты памяти один web-процесс одновременно обрабатывает не более двух документов; внутри одного пользовательского пакета файлы обрабатываются последовательно.

## Переменные Render

В Render → Web Service → Environment добавьте или проверьте:

```text
ENABLE_DOCUMENT_ANALYSIS=true
OPENAI_API_KEY=ваш_секретный_API_ключ
OPENAI_DOCUMENT_MODEL=модель_с_поддержкой_изображений_и_PDF
DOCUMENT_ANALYSIS_MAX_OUTPUT_TOKENS=6000

Для `gpt-5-mini` версия 3.5.9 автоматически использует низкий уровень reasoning и низкую verbosity. Это важно: `max_output_tokens` включает не только видимый JSON, но и reasoning-токены. Если ответ всё же обрезан по лимиту, приложение один раз повторит запрос с увеличенным бюджетом и запишет точную причину в журнал.
DOCUMENT_ANALYSIS_TIMEOUT_SECONDS=90
```

Если `OPENAI_DOCUMENT_MODEL` оставить пустой, используется `OPENAI_MODEL`.

Существующие переменные `DATABASE_URL`, `APP_SECRET`, `ADMIN_TOKEN`, почтовые настройки и другие секреты сохраняйте без изменений.

## Проверка после развёртывания

Откройте:

```text
https://chinatraderesolve.onrender.com/health
```

Ожидаемое поле:

```json
{
  "document_analysis_enabled": true
}
```

Затем:

1. отправьте тестовую заявку с включённым согласием на ИИ;
2. откройте закрытую ссылку статуса;
3. загрузите один тестовый PNG или PDF без реальных персональных данных;
4. нажмите «Проанализировать документы с помощью ИИ»;
5. убедитесь, что появился отчёт;
6. войдите в `/admin` и проверьте, что файл и отчёт видны в карточке дела.

## Безопасность и приватность

- не загружайте `.env`, API-ключи и пароли в GitHub;
- не просите клиентов загружать пароли, seed-фразы, приватные ключи, полные номера банковских карт и ненужные документы личности;
- ссылка статуса является секретной ссылкой доступа к делу — её нельзя публиковать;
- загруженные документы передаются настроенному ИИ-провайдеру только после нажатия анализа и при ранее данном согласии на ИИ;
- запросы к Responses API отправляются с `store: false`;
- закрытые дела, файлы и отчёты удаляются или обезличиваются политикой хранения, по умолчанию через 90 дней;
- важные выводы всегда требуют проверки человеком.
- зашифрованные PDF, PDF со встроенными файлами, JavaScript, launch-действиями, XFA или rich media отклоняются до сохранения;
- администраторские действия защищены CSRF-токеном сессии;
- один и тот же случай нельзя одновременно отправить на два анализа.

## Модель выполнения и ограничение текущей версии

После нажатия сервер атомарно закрепляет за делом единственную задачу анализа, сразу возвращает страницу и продолжает обращение к OpenAI в фоне внутри веб-процесса. Страница автоматически проверяет статус. Пока анализ идёт, документы нельзя добавлять или удалять, поэтому отчёт относится к неизменному набору доказательств.

Каждый запуск получает уникальный внутренний токен владения. Сохранить результат или ошибку может только worker текущего запуска, поэтому поздний ответ старой задачи не перезапишет новый повторный анализ. При zero-downtime деплое свежая работа не сбрасывается: приложение помечает неудачными только задачи, которые остаются в состоянии `running` дольше рассчитанного безопасного таймаута. Для высокой нагрузки и гарантированной доставки заданий следующим архитектурным шагом остаётся отдельная очередь и worker.


## Поведение согласия в v3.4.2

Если пользователь не отметил согласие на ИИ при первоначальной заявке, загрузка файлов всё равно доступна.
Перед запуском анализа закрытая страница дела отдельно покажет обязательную галочку добровольного согласия.
Администратор не может запустить анализ без этого согласия.

При ошибке провайдера в журнале дела появится событие «ИИ-анализ документов завершился ошибкой»
с безопасной диагностикой: тип подключения, HTTP-статус и идентификатор запроса OpenAI, если он был возвращён.
Секретный API-ключ в журнал не записывается.

## Проверка ссылок на доказательства

Перед сохранением отчёта версия 3.5.9 сверяет названия файлов, указанные ИИ, с фактически загруженными документами. Вымышленные имена и повторяющиеся ссылки отбрасываются; событие хронологии без хотя бы одного реального файла-источника не показывается как доказанный факт.
